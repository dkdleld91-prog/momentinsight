# Project Memory

## 제품 구조

- 화면 원본: `src/pages/home.html`, `src/pages/admin.html`, `src/pages/client.html`
- 서버 API: `src/server/handlers`
- 배포 API 진입점: `api`
- 운영 데이터: Supabase와 서버 측 권한 검증을 사용한다.
- 배포 대상: Vercel Production `https://insight.momentlabs.co.kr`

## 고정 권한 기준

- 총관리자 계정은 `mml93-a01`이다.
- 운영팀은 연결된 광고주 1명의 데이터만 관리한다.
- 광고주는 본인 코드에 연결된 공개 데이터만 확인한다.
- 내부 메모와 광고주 공개 코멘트는 분리한다.
- 화면 숨김만으로 권한을 판단하지 않고 서버와 DB 범위를 함께 검증한다.

## 네이버 상품 순위 고정 기준

- 공식 쇼핑 검색 API에서 광고를 제외한 정확도순 오가닉 순서를 `네이버쇼핑 오가닉 순위`로 사용한다. 임의 중복 제거로 순서를 바꾸지 않는다.
- 화면의 페이지 표기는 네이버 가격비교 `40개씩 보기` 기준이다. `page=ceil(rank/40)`, `position=((rank-1)%40)+1`, `pageSize=40`을 서버·관리자·광고주·30일 기록에 동일 적용한다.
- 판매자 상품 URL은 결과 링크 `/products/{id}`의 판매자 상품ID를 최우선 정확 일치 키로 사용한다. API `productId`와 판매자 상품ID를 같은 값으로 취급하지 않는다.
- 정확 상품을 찾은 뒤에도 선택 범위 300개까지 계속 확인해 관련 원부와 누락 후보를 함께 수집한다.
- 관련 원부는 키워드, 브랜드·제조사·판매처 중 하나, 상위 카테고리 2단계가 모두 맞는 가격비교 원부 중 가장 높은 1건만 인정한다.
- 단건과 30일 추적 모두 정확 상품과 검증된 관련 원부 중 숫자가 더 낮은 오가닉 결과를 대표 `rank`, `page`, `position`, 상품 형태로 사용한다. 원부가 높으면 `원부형`, 정확 상품이 높거나 같으면 해당 상품 형태를 표시한다.
- 대표값이 원부여도 `targetProductId`, 입력 URL, 정확 상품 판매자ID는 변경하지 않는다. `representativeItem`과 `exactItem`을 분리하고 카드 2건을 함께 제공한다.
- 같은 원부의 다른 판매자 상품, 같은 판매처의 무관 상품, 브랜드·카테고리가 다른 원부는 정확 상품 또는 대표 원부로 처리하지 않는다.
- 상품명은 조회 키워드의 네이버쇼핑 검색 결과로, `상품 열기`는 원부 카탈로그 또는 사용자가 입력한 정확 상품 URL로 연결한다.
- 관련 원부 배지는 네이버 민트·그린, 정확 상품은 딥네이비를 사용하고 두 역할에서 동일한 컴팩트 카드 레이아웃을 유지한다.
- 공식 API가 광고 목록과 위치를 제공하지 않으므로 광고 순위·광고 개수는 임의 계산하지 않고 `광고상품 미연결`로 남긴다.
- 로그인된 네이버 가격비교 화면에서 `전동칫솔` 원부 34위(1페이지 34위), 정확 상품 `12649811979` 168위(5페이지 8위)와 API 오가닉 순서가 일치함을 확인했다. 독립 서버의 화면 자동 접근은 HTTP 418이므로 성공한 것처럼 주장하지 않는다.
- 과거 30일 기록은 당시 원부 순번 원본이 없으면 소급 변경하지 않고 새 갱신부터 대표 기준을 적용한다.
- 네이버 Developers 쇼핑 검색 API는 2026-07-31 종료 예정이므로, 종료 전에 공식 대체 엔드포인트를 확인해 같은 오가닉 순위 계약을 유지한다.

## 보호 기능

- 키워드 조회, 네이버 SEO 확인, 네이버 상품 단건 오가닉 순위
- 네이버 상품 30일 대표 순위, 네이버 플레이스 30일 순위
- 오전 9시·오후 3시 자동 갱신과 최근 30일 기록
- 관리자·운영팀·광고주 로그인과 데이터 격리
- 운영 원본, 보고서 공개, 일정과 액션 플랜

## 네이버 플레이스 순위 고정 기준

- 플레이스 ID가 URL에 있으면 상호명 유사도보다 ID 일치를 우선한다.
- Apify 결과는 `organicRank`, `searchRank`, `rank`, `position`을 우선해 정렬하고 광고·중복을 제거한다.
- Apify 계정 한도 초과 시 같은 계정의 다른 Actor를 반복 호출하지 않고 자체 브라우저 수집기로 전환한다.
- `checkedCount < 300`인 미발견 결과는 300위 밖으로 확정하지 않고 확인 범위 이후를 미검증으로 남긴다.

## 검증 원칙

현황 확인 → 관련 코드 분석 → 구현 → 자동 테스트 → 실조회 → 데스크톱·모바일 UI 검수 → 빌드 → 문서 갱신 → 커밋 → 승인 시 배포 순서를 지킨다.

## 지속 개발 운영 계약

- 시작할 때 Git 상태, 최근 커밋, 작업명세, 현재 상태, 다음 작업을 읽고 `npm run work:autosave`를 실행한다.
- 기존 구현·문서·테스트를 먼저 검색해 같은 기능과 파일을 다시 만들지 않는다.
- 사용자의 미커밋 변경과 관련 없는 diff는 보존한다. 특히 `admin.html`과 `client.html`은 한쪽 변경이 다른 쪽의 기능·권한·레이아웃을 깨지 않는지 함께 확인한다.
- 변경은 작고 되돌릴 수 있는 논리 단위로 나누며, 각 단위의 목적·검증·결과를 작업명세와 운영 문서에 남긴다.
- 완료 전 대상 테스트, 다른 관점의 2차 검증, `git diff --check`, 최종 diff와 상태 확인을 수행한다.
- 검증된 논리 단위마다 로컬 커밋해 다음 작업의 기준점을 만든다. 비밀값, 임시 파일, 재생성 산출물은 커밋하지 않는다.
- 외부 한도나 계정 문제로 핵심 실검증이 불가능하면 보류 상태와 재개 조건을 기록하며, 완료·정상화·배포로 처리하지 않는다.
- push와 배포는 현재 작업에서 명시된 승인 범위 안에서만 수행하고, 승인 이후에도 필수 검증 실패 시 중단한다.

## 역할 공통 사이드바 UI 기준

- 운영팀과 광고주 사이드바는 `운영 → 키워드·SEO → 순위 조회·추적 → 광고 조사` 네 개 메뉴 분류를 공통으로 사용한다.
- 역할별 기능과 권한은 유지하되 사이드바 폭, 브랜드 마크, 그룹 제목, 메뉴 간격, hover·active 상태, 인증 패널의 시각 체계는 동일하게 유지한다.
- 광고주는 운영 기능을 추가로 받는 것이 아니라 본인에게 허용된 대시보드, 매출, 일정, 대행사 연결과 마케팅 도구만 공통 분류 안에 표시한다.
- 데스크톱은 고정 사이드바, 모바일은 상단 가로 스크롤 메뉴를 사용하며 390px에서 문서 전체 가로 넘침이 없어야 한다.
- `data-mi-shell="premium-sidebar"`와 릴리즈 기준선 검사로 두 역할의 공통 쉘 누락을 방지한다.

## 대시보드 상태 진실성 기준

- 운영 홈에는 DB·API에서 실제 집계하지 않은 광고주 수, 누락 건수, 승인 대기 건수나 예시 브랜드를 운영 수치처럼 표시하지 않는다.
- 실집계가 연결되기 전에는 계정 연결, 운영 입력, 공개 승인, 보고서 관리로 이동하는 작업 카드만 제공한다.
- 광고주 공개 데이터가 비어 있으면 `업데이트 완료`, `검수 완료`, `공개용`으로 표시하지 않고 `공개 준비 중`, `공개 데이터 연결 대기`, `데이터 대기`로 명확히 구분한다.
- 빈 공개 상태에서 기본 데모 보고서가 섞이지 않도록 현재 `reportCenterSynced` 상태를 보고서 렌더러에 직접 전달한다.
- 운영팀·광고주 메뉴 전환 시 이전 화면의 스크롤 위치를 유지하지 않고 새 화면의 상단 제목부터 보여준다.
- 390px 광고주 빈 상태는 핵심 지표 3개와 오늘 결론·다음 행동을 한 화면에 가깝게 압축하고 문서 가로 넘침이 없어야 한다.
