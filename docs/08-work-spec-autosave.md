# 작업 명세서 및 오토세이브 관리

이 문서는 모먼트 인사이트 개발 작업의 기준 문서입니다.
앞으로 새 기능을 만들거나 기존 기능을 수정할 때는 이 문서에 작업 의도, 실행 내역, 검증 결과를 남기고 개발 완료 시 체크합니다.

## 운영 규칙

- 개발 착수 전: 작업 항목을 추가하고 상태를 `진행 중`으로 둡니다.
- 개발 완료 후: 작업 항목을 `완료`로 체크하고 검증 결과를 적습니다.
- 데이터 관리는 로컬 `git commit`까지만 진행합니다.
- `git push`, Vercel 배포, Supabase 배포는 대표님이 명시적으로 지시할 때만 진행합니다.
- 배포 산출물(`dist`, 임시 캐시)은 필요 시 재생성하고 기본적으로 저장소에 남기지 않습니다.
- API 키가 들어가는 `.env.local` 파일은 삭제하거나 커밋하지 않습니다.

## 작업 상태 기준

- `대기`: 아직 시작하지 않은 작업
- `진행 중`: 코드 또는 문서 작업 중
- `완료`: 개발과 검증이 끝나 로컬 커밋된 작업
- `보류`: 외부 키, 계정, 정책, 운영 판단이 필요한 작업
- `배포 대기`: 커밋은 완료됐지만 대표님 배포 지시 전 상태

## 프로젝트 고정 기억

앞으로 작업을 이어갈 때 아래 기준은 다시 흔들지 않는다. 새 기능을 추가하더라도 기존 로그인, 권한, 데이터 격리, 순위추적, 보고서 운영 흐름을 깨지 않는 범위에서만 수정한다.

### 계정과 권한

- 총관리자 계정은 `mml93-a01`이다.
- 총관리자는 운영팀 코드 생성, 광고주 직접 생성, 활성 계정 조회, 권한 해제를 담당한다.
- 운영팀은 계정당 광고주 1명만 연결한다.
- 광고주는 운영팀 또는 총관리자가 발급한 광고주 코드로만 접속한다.
- 광고주 화면에는 운영팀 코드, 내부 메모, API 기준, 비밀값, 내부 오류 상세를 노출하지 않는다.
- 로그아웃 기능은 유지하고, 새로고침 시에는 저장된 코드로 서버 재검증 후 세션을 복구한다.

### 현재 사용 가능 도구

- 현재 안정 사용 기능은 키워드 조회, 네이버 SEO 확인, 네이버 상품 순위 조회, 네이버 30일 순위 추적이다.
- Meta 광고 조사는 메뉴와 화면에서 `개발중`으로 표시한다.
- 네이버 플레이스 순위 추적은 상품 순위와 분리된 별도 기능으로 다룬다.
- 외부 API 기준과 내부 계산식은 광고주 화면에 설명하지 않는다.

### 네이버 순위 추적

- 네이버 상품 순위는 광고를 제외한 오가닉 기준으로 계산한다.
- 상품 URL이 원부형이면 원부 묶음에 포함된 판매자는 같은 원부 순위로 취급한다.
- 30일 순위 추적은 30일 후 중단하지 않는다. 계속 수집하고 화면에는 최근 30일만 보여준다.
- 자동 갱신 기준은 매일 오전 9시와 오후 3시다.
- 자동 갱신 누락 방지를 위해 GitHub Actions와 Vercel 백업 Cron을 함께 점검한다.
- 선택 이미지 저장, 그룹 보기, 그룹 생성 및 이동, 그룹 해제는 기존 추적 데이터가 날아가지 않도록 보존 우선으로 수정한다.

### 네이버 API 기준

- 기존 네이버 Developers 앱에는 `검색`, `데이터랩(검색어트렌드)`, `데이터랩(쇼핑인사이트)`가 등록되어 있다.
- 키워드 조회와 상품 순위 관련 환경변수는 배포 환경변수로만 관리하고, 키 값은 코드와 문서에 남기지 않는다.
- 네이버 플레이스 300위 전체 추적은 공식 검색 API만으로 완전 구현되지 않는다. 필요 시 외부 순위 수집 API 또는 별도 수집 서버를 연결한다.

### 보고서와 운영 입력

- 주간보고서는 운영 부담이 크므로 기본 운영 범위에서 제외한다.
- 운영팀은 월간 기준으로 매출, 광고비, 노출, 클릭, 전환율, 전체 구매수, 전체 매출을 입력한다.
- 월간 합계는 일별 입력에서 자동 집계되도록 설계한다.
- 보고서는 엑셀 원본 입력값과 운영팀이 작성한 로드맵/액션플랜을 바탕으로 PPTX 또는 PDF 산출물로 확장한다.
- 로드맵과 액션플랜은 AI가 임의로 계산하는 값이 아니라 운영팀이 직접 작성하는 운영 판단 영역이다.

### UI/UX 기준

- 전체 톤은 딥네이비, 화이트, 그레이 중심의 B2B SaaS 프리미엄 톤을 유지한다.
- 글이 많아지면 먼저 줄이고, 다음으로 접기/더보기/상태 배지로 정리한다.
- 긴 상품명, 코드, 검색량, 순위는 한 줄 정렬과 말줄임을 우선하고 레이아웃 겹침이 생기면 배포하지 않는다.
- 버튼 hover, active, disabled, focus-visible 상태는 기본 품질로 유지한다.
- 카카오 문의 버튼과 개발중 안내 배너는 안내형 톤으로 두고, 위협적이거나 저렴해 보이는 표현은 피한다.

### 검증과 배포

- 성공 보고 전에는 `work:autosave`, 서버 문법 검사, 빌드, 핵심 API 또는 화면 마커 검증을 실행한다.
- 검증하지 못한 항목은 성공으로 보고하지 않는다.
- 대표님이 배포 승인한 경우에만 push와 배포를 진행한다.
- 배포 전후로 `git status`, 최신 커밋, 라이브 `/health`, 핵심 API 200/401 응답을 확인한다.
