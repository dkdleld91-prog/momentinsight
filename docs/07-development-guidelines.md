# 개발 시 주의사항

## 권한과 보안

- 화면에서 메뉴를 숨기는 것만으로 권한을 처리하지 않는다.
- API와 DB 조회 단계에서 `client_id`, `brand_id`, 역할을 검증한다.
- 내부 메모는 서버 응답에서 광고주 사용자에게 내려주지 않는다.
- RLS는 행 접근 제어이므로 `internal_note` 같은 민감 컬럼은 client-safe view 또는 명시적 컬럼 select로 제외한다.
- 광고주 API에서는 `.select("*")`를 금지한다.
- 파일 URL은 공개 링크와 비공개 스토리지 접근 방식을 구분한다.
- 관리자 역할도 담당 범위를 제한할 수 있게 설계한다.

## 데이터 계산

- ROAS, CTR, CVR, CPA, CPC 같은 계산식은 한곳에서 관리한다.
- 0으로 나누는 상황을 방어한다.
- 원화, 퍼센트, 순위, 건수 포맷을 일관되게 처리한다.
- 수동 입력값과 자동 연동값의 출처를 구분한다.

## 운영 UX

- 관리자는 광고주를 먼저 선택하고 작업하게 한다.
- 광고주 화면은 읽기 중심으로 단순하게 유지한다.
- 필터 기본값은 현재 월과 현재 브랜드로 둔다.
- 빈 상태에서는 다음 입력 행동을 바로 안내한다.
- 보고서와 일정은 날짜 기준 필터가 가장 중요하다.

## 문서와 코드 경계

- `docs/`는 기획과 설계 기준이다.
- `database/`는 DB 설계 초안과 마이그레이션 이전 문서를 둔다.
- `src/features/`는 기능 도메인별 구현을 둔다.
- `src/components/`는 도메인에 묶이지 않는 공통 UI만 둔다.
- 계산 로직은 컴포넌트 내부보다 `src/lib` 또는 기능별 유틸에 둔다.

## 네이밍 규칙

- 데이터베이스는 snake_case를 사용한다.
- TypeScript 타입과 컴포넌트는 PascalCase를 사용한다.
- 함수와 변수는 camelCase를 사용한다.
- 화면 경로는 kebab-case를 사용한다.

## 1차 구현에서 피해야 할 것

- 자동 연동을 먼저 만들지 않는다.
- 보고서 생성 AI 기능을 MVP 핵심에 넣지 않는다.
- 광고 플랫폼별 복잡한 예외를 초기 DB에 과도하게 반영하지 않는다.
- 모든 화면을 한 번에 완성하려 하지 않는다.
- 관리자 입력 흐름 없이 광고주 대시보드만 먼저 만들지 않는다.

## 품질 체크리스트

- 광고주 계정으로 다른 광고주 데이터 접근이 차단되는가
- 내부 메모가 광고주 화면과 API 응답에서 제외되는가
- 내부 메모가 Supabase 직접 조회에서도 광고주 권한으로 노출되지 않는가
- KPI 달성률 계산이 목표값 0일 때 깨지지 않는가
- 파일과 보고서가 공개 여부에 따라 필터링되는가
- 모바일에서 표와 카드가 겹치지 않는가
- 빈 데이터 상태에서도 화면이 이해 가능한가
