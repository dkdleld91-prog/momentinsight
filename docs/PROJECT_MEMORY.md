## 네이버 상품 순번 고정 기준

- 현재 숫자는 네이버 쇼핑 검색 API의 정확도순 결과 배열 순번이다. 실제 네이버 쇼핑 화면 순위 또는 광고 제외 오가닉 페이지 순위라고 부르지 않는다.
- API 배열 순번을 40개 단위로 나눠 `N페이지 N위`로 환산하지 않는다. 서버의 `page`, `position`, `pageSize`는 실화면 수집 근거가 없으면 `null`이다.
- 판매자 상품ID가 정확히 같다는 사실은 `상품 ID 일치`로 표시하며, 실화면에 같은 위치로 노출됐다는 뜻의 `정확 상품 순위`로 표현하지 않는다.
- 판매자 상품 URL은 URL의 판매자 상품ID를 최우선 정확 일치 키로 사용한다.
- 판매자 상품 조회에서는 네이버 검색 API의 `productId`와 결과 링크의 판매자 상품ID를 같은 값으로 취급하지 않는다. 정확 일치는 결과 링크 `/products/{id}`의 판매자 상품ID 또는 완전한 정규 URL만 사용한다.
- 상품 ID 일치를 먼저 찾더라도 선택한 조회 범위까지 API 결과를 계속 확인해 `checkedCount`와 관련 노출 후보가 중간에서 끊기지 않게 한다.
- 네이버 공식 API의 결과 배열은 임의 중복 제거하지 않는다. 동일해 보이는 항목을 제거하면 API 순번이 바뀌므로 제공 순서를 그대로 API 결과 근거로 사용한다.
- 판매자 상품 URL을 제목, 카테고리 또는 알려진 별칭만으로 원부 카탈로그에 자동 연결하지 않는다.
- 같은 원부라도 상품ID 또는 판매자가 다르면 정확 상품으로 처리하지 않는다.
- 원부 카탈로그 URL이나 명시적 원부 ID가 입력된 경우에만 원부 모드를 사용한다.
- 판매자 상품 조회에서는 상품 ID 일치와 관련 원부 결과를 분리한다. 관련 원부는 키워드, 브랜드·제조사·판매처 중 하나, 상위 카테고리 2단계가 모두 맞는 API 결과 중 가장 앞선 1건만 표시한다.
- 단건 조회에서 관련 원부는 화면 참고 정보이며 대상 상품의 API 결과 순번과 `targetProductId`를 변경하지 않는다.
- 30일 추적은 예외적으로 정확 상품과 검증된 관련 원부의 공식 API 순번 중 숫자가 더 낮은 쪽을 대표값으로 저장한다. 대상 상품ID는 그대로 유지하고 스냅샷에 `상품 기준` 또는 `원부 기준` 출처를 함께 남긴다.
- 기존 30일 기록은 당시 관련 원부 순번 원본이 없어 소급 변경하지 않으며 새 기준 적용 후 생성되는 스냅샷부터 반영한다.
- 단건 상품 노출 카드는 운영팀과 광고주에서 동일한 컴팩트 레이아웃을 사용한다. 상품명은 조회 키워드의 네이버 쇼핑 검색 결과로, `상품 열기`는 해당 상품 상세 URL로 연결한다.
- 정확 상품 카드의 `상품 열기`는 검색 API가 변환한 스토어 주소가 아니라 사용자가 입력한 원본 네이버 상품 URL을 유지한다.
- 관련 원부 배지는 갈색 대신 네이버 계열의 옅은 민트·그린 톤을 사용하며 상품 ID 일치의 딥네이비 배지와 구분한다.
- 공식 API가 광고 목록과 광고 위치를 제공하지 않으므로 광고 순위나 광고 개수를 임의 계산하지 않는다.
- 단순히 판매처만 같은 다른 상품은 상품 노출 카드에서 제외한다.
- 30일 추적의 대표 상품ID는 저장된 URL에서 다시 추출해 오래된 카탈로그 ID로 오염되지 않게 한다.
- 네이버 Developers 쇼핑 검색 API는 2026-07-31 종료 예정이므로, 종료 전에 NAVER API HUB 이관 여부와 실제 화면 순위 수집 정책을 별도 확정한다.

## 네이버 플레이스 순위 고정 기준

- 플레이스 ID가 URL에 있으면 상호명 유사도보다 ID 일치를 우선한다.
- Apify 결과는 `organicRank`, `searchRank`, `rank`, `position`을 우선해 정렬하고 광고·중복을 제거한다.
- Apify 계정 한도 초과 시 같은 계정의 다른 Actor를 반복 호출하지 않고 자체 브라우저 수집기로 전환한다.
- `checkedCount < 300`인 미발견 결과는 300위 밖으로 확정하지 않고 확인 범위 이후를 미검증으로 남긴다.

## 지속 개발 운영 계약

- 시작할 때 Git 상태, 최근 커밋, 작업명세, 현재 상태, 다음 작업을 읽고 `npm run work:autosave`를 실행한다.
- 기존 구현·문서·테스트를 먼저 검색해 같은 기능과 파일을 다시 만들지 않는다.
- 사용자의 미커밋 변경과 관련 없는 diff는 보존한다. 특히 `admin.html`과 `client.html`은 한쪽 변경이 다른 쪽의 기능·권한·레이아웃을 깨지 않는지 함께 확인한다.
- 변경은 작고 되돌릴 수 있는 논리 단위로 나누며, 각 단위의 목적·검증·결과를 작업명세와 운영 문서에 남긴다.
- 완료 전 대상 테스트, 다른 관점의 2차 검증, `git diff --check`, 최종 diff와 상태 확인을 수행한다.
- 검증된 논리 단위마다 로컬 커밋해 다음 작업의 기준점을 만든다. 비밀값, 임시 파일, 재생성 산출물은 커밋하지 않는다.
- 외부 한도나 계정 문제로 핵심 실검증이 불가능하면 보류 상태와 재개 조건을 기록하며, 완료·정상화·배포로 처리하지 않는다.
- push와 배포는 현재 작업에서 명시된 승인 범위 안에서만 수행하고, 승인 이후에도 필수 검증 실패 시 중단한다.

## 역할 공통 사이드바 UI 기준

- 운영팀과 광고주 사이드바는 `운영 → 키워드·SEO → 순위 조회·추적 → 광고 조사` 네 개 메뉴 분류를 공통으로 사용한다.
- 역할별 기능과 권한은 유지하되 사이드바 폭, 브랜드 마크, 그룹 제목, 메뉴 간격, hover·active 상태, 인증 패널의 시각 체계는 동일하게 유지한다.
- 광고주는 운영 기능을 추가로 받는 것이 아니라 본인에게 허용된 대시보드, 매출, 일정, 대행사 연결과 마케팅 도구만 공통 분류 안에 표시한다.
- 데스크톱은 고정 사이드바, 모바일은 상단 가로 스크롤 메뉴를 사용하며 390px에서 문서 전체 가로 넘침이 없어야 한다.
- `data-mi-shell="premium-sidebar"`와 릴리즈 기준선 검사로 두 역할의 공통 쉘 누락을 방지한다.

## 대시보드 상태 진실성 기준

- 운영 홈에는 DB·API에서 실제 집계하지 않은 광고주 수, 누락 건수, 승인 대기 건수나 예시 브랜드를 운영 수치처럼 표시하지 않는다.
- 실집계가 연결되기 전에는 계정 연결, 운영 입력, 공개 승인, 보고서 관리로 이동하는 작업 카드만 제공한다.
- 광고주 공개 데이터가 비어 있으면 `업데이트 완료`, `검수 완료`, `공개용`으로 표시하지 않고 `공개 준비 중`, `공개 데이터 연결 대기`, `데이터 대기`로 명확히 구분한다.
- 빈 공개 상태에서 기본 데모 보고서가 섞이지 않도록 현재 `reportCenterSynced` 상태를 보고서 렌더러에 직접 전달한다.
- 운영팀·광고주 메뉴 전환 시 이전 화면의 스크롤 위치를 유지하지 않고 새 화면의 상단 제목부터 보여준다.
- 390px 광고주 빈 상태는 핵심 지표 3개와 오늘 결론·다음 행동을 한 화면에 가깝게 압축하고 문서 가로 넘침이 없어야 한다.
