# AGENTS.md

이 프로젝트는 마케팅 대행사용 B2B SaaS 대시보드입니다. 모든 개발과 문서화는 광고주별 데이터 격리, 관리자 운영 효율, 대시보드 가독성, 실행 인사이트 제공을 우선합니다.

## 서비스 목표

모먼트 인사이트는 단순 자료 업로드 공간이 아니라 광고주가 자기 브랜드 성과, 현재 마케팅 진행 상황, 향후 실행 방향을 한곳에서 이해하는 통합 마케팅 관리 플랫폼입니다.

광고주 화면은 `현재 상태 요약 -> 핵심 지표 -> 채널별 성과 -> 분석 인사이트 -> 다음 실행 계획 -> 일정 및 요청사항` 흐름으로 설계합니다. 운영팀/총관리자 화면은 입력, 검수, 공개, 권한, 파일, 보고서 관리를 빠르게 처리하는 운영 콘솔로 설계합니다.

## 제품 원칙

- 광고주는 본인 브랜드 데이터만 볼 수 있어야 한다.
- 관리자는 전체 광고주, 브랜드, KPI, 보고서, 일정, 파일을 통합 관리할 수 있어야 한다.
- 내부 메모와 광고주 공개 코멘트는 데이터 구조와 UI에서 반드시 분리한다.
- 대시보드는 숫자 나열이 아니라 `현재 상태 요약 -> 핵심 지표 -> 채널별 성과 -> 인사이트 -> 다음 액션` 흐름을 따른다.
- 1차 MVP는 수동 입력과 업로드 기반의 운영 가능성을 우선한다.
- 광고주가 보는 수치는 운영팀/총관리자가 공개한 데이터만 사용한다.
- 운영팀 1개 계정은 광고주 1개 연결을 기본으로 유지한다.
- 총관리자 `mml93-a01`은 운영팀/광고주 코드 발급과 권한 해제를 관리하는 최상위 계정이다.
- 외부 API 기준, 내부 계산 방식, 비밀값은 광고주 화면에 노출하지 않는다.

## 권한 구조

- `owner`: 총관리자. 운영팀 코드 생성, 광고주 직접 생성, 활성 계정 조회, 권한 해제, 제한 예외를 가진다.
- `operator`: 운영팀. 연결된 광고주 1명의 공개 데이터, 보고서, 일정, 파일, 순위추적을 관리한다.
- `client`: 광고주. 자기 코드로 연결된 공개 데이터, 보고서, 일정, 도구 결과만 확인한다.

권한 체크는 UI 숨김으로 끝내지 않고 서버 요청과 DB 스코프에서 함께 검증합니다. 모든 주요 데이터는 `client_id`, 운영팀 코드, 또는 광고주 코드 기준으로 분리합니다.

## 핵심 화면

- 홈페이지: 모먼트 인사이트와 마케팅 사이트의 신뢰를 전달하고 광고주/운영팀 진입을 안내한다.
- 광고주 대시보드: 매출, 광고비, ROAS, 목표 달성률, 일정, 보고서, 실행 인사이트를 요약한다.
- 운영팀 콘솔: 광고주 공개 데이터 입력, 보고서 업로드, 일정 관리, 파일 관리, 도구 실행, 공개 상태 확인을 제공한다.
- 총관리자 콘솔: 운영팀/광고주 코드 발급, 활성 계정 관계도, 권한 해제, 운영 헬스체크를 제공한다.
- 도구: 키워드 조회, 네이버 상품 순위 조회, 네이버 30일 순위 추적, 네이버 SEO 확인을 분리해 제공한다.

## 데이터 범위

- 대시보드: 월 누적 매출, 광고비, ROAS, 전환율, 클릭률, 노출수, 구매수량, 리뷰수, 전월 대비, 목표 대비 달성률.
- KPI: 목표 매출, 실제 매출, 광고비, ROAS, 구매 수량, CPA, CPC, CTR, CVR, 리뷰 수, 키워드 순위, 검색 노출.
- 광고 성과: 네이버, 쿠팡, 메타, 자사몰, 검색광고, 디스플레이 광고별 광고비, 매출, ROAS, 노출, 클릭, 전환, 소재/캠페인 성과.
- 키워드/순위: 검색량, 오가닉 순위, 이전 순위, 변화, SEO 보완 필요 여부, 실행 인사이트.
- 보고서/파일: 주간, 월간, KPI, 매출, 광고 성과, 키워드 분석, 캠페인 결과, 촬영/콘텐츠 보고서와 원천 파일.
- 일정: 광고 세팅, 콘텐츠 업로드, 배포, 리뷰, 촬영, 프로모션, 보고서 제출, 미팅, 소재 제작, 키워드 작업.

## UI 원칙

- 전문적인 B2B SaaS 톤을 유지한다.
- 컬러는 블랙, 화이트, 그레이, 딥네이비를 중심으로 사용한다.
- 핵심 수치는 크게, 상세 데이터는 표로 정돈한다.
- 상태 배지, 진행률 바, 카드, 차트, 테이블을 일관되게 사용한다.
- 광고주 화면은 설명 없이도 현재 성과와 다음 할 일을 이해할 수 있어야 한다.
- 카드와 표는 여백을 과하게 쓰지 않고, 화면별 첫 시야에서 핵심 지표가 바로 보여야 한다.
- CTA는 hover, active, focus-visible, disabled 상태를 명확히 제공한다.
- 긴 상품명, 코드, 검색량은 잘리지 않도록 최소 폭과 말줄임 기준을 함께 둔다.
- 광고주 화면에는 내부 메모, API 기준, 운영 코드, 비밀값, 내부 오류 상세를 노출하지 않는다.

## 개발 원칙

- 기능은 광고주 워크스페이스 단위로 설계한다.
- 모든 주요 테이블은 `client_id` 또는 브랜드 스코프를 명확히 가진다.
- 권한 체크는 화면 숨김만으로 처리하지 않고 서버 데이터 접근 단계에서 검증한다.
- 보고서, 파일, 링크는 같은 자료 관리 모델에서 다룬다.
- 수치 계산 로직은 UI 컴포넌트에 직접 흩뿌리지 않고 도메인 유틸 또는 서버 계층에 둔다.
- 순위추적은 광고 제외 오가닉 기준으로 계산한다.
- 순위추적 기록은 30일 뒤 중단하지 않고 계속 수집하되 화면에는 최근 30일만 표시한다.
- 네이버 30일 순위 자동 수집 기준은 매일 오전 9시와 오후 3시다.
- 데모값은 운영 화면에 기본값처럼 보이게 만들지 않는다. 실제 연결 전에는 미연결/입력 필요 상태로 처리한다.
- API 키, 토큰, 비밀값은 `.env` 또는 배포 환경변수로만 다루고 커밋하지 않는다.
- 새 기능은 기존 광고주/운영팀 데이터 격리를 깨지 않는지 반드시 검증한다.

## 지속 개발 및 충돌 방지 원칙

- 작업 시작 전 `git status --short --branch`, 최근 커밋, `docs/08-work-spec-autosave.md`, `docs/WORK_STATUS.md`, `docs/NEXT_ACTIONS.md`를 확인한다.
- `npm run work:autosave`를 실행하고 작업명세서에 이번 작업의 목적, 범위, 상태를 먼저 기록한다.
- 새 파일이나 로직을 만들기 전에 `rg`로 기존 구현, 문서, 테스트를 검색해 중복 개발을 막는다.
- 사용자의 미커밋 변경과 현재 브랜치의 기존 작업은 임의로 되돌리거나 덮어쓰지 않는다. 범위가 겹치면 현재 diff를 먼저 읽고 최소 단위로 합친다.
- `src/pages/admin.html`과 `src/pages/client.html` 중 하나를 바꾸면 다른 화면의 공통 기능과 권한 차이를 함께 확인하고 두 화면을 모두 검증한다.
- 한 커밋에는 하나의 논리적 작업만 담는다. 관련 없는 변경, 재생성 가능한 산출물, 비밀값을 섞지 않는다.
- 완료 또는 외부 차단이 확인되면 작업명세, `PROJECT_MEMORY`, `WORK_STATUS`, `DECISIONS`, `NEXT_ACTIONS`, `TEST_EVIDENCE` 중 해당 문서를 같은 작업에서 갱신한다.
- 커밋 전 대상 테스트, 다른 관점의 2차 검사, `git diff --check`, 최종 diff와 상태 확인을 수행한다.
- 강제 리셋, 이력 재작성, 무단 push는 금지한다. 배포는 현재 작업의 승인과 필수 검증이 모두 충족된 경우에만 진행한다.
- 외부 서비스 한도나 계정 상태로 실검증이 막히면 `보류` 또는 `배포 대기`로 남기고 완료나 정상화를 과장하지 않는다.

## MVP 우선순위

1. 광고주별 대시보드, KPI 관리, 보고서 업로드, 일정표, 관리자 기능.
2. 채널별 광고 성과 분석, 키워드 분석, 자동 그래프, 인사이트 입력.
3. 네이버, 쿠팡, 메타, 구글시트, 엑셀 데이터 연동.

새 개발은 위 순서를 기준으로 하되, 운영 중인 로그인, 권한, 보고서, 순위추적 기능을 깨뜨리지 않는 작은 단위로 배포합니다.

## 검증 원칙

- 작업 전 `npm run work:autosave`로 작업명세를 갱신한다.
- 서버/HTML/빌드 검증을 최소 1회 이상 실행한다.
- 사용자에게 성공을 보고하기 전에 같은 핵심 검증을 2차로 반복하거나 다른 관점의 검증을 추가한다.
- 검증하지 못한 항목은 성공으로 말하지 않는다.
- 배포는 대표님이 명시적으로 요청했거나, 현재 대화에서 배포 승인을 준 경우에만 진행한다.
